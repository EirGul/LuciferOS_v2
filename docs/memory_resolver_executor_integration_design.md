# Memory Resolver to Executor Integration Design

## Milestone 112 status

Milestone 112 is design-only.

It must not connect MemoryTargetResolver to MemoryCommandExecutor, IntentRouter, Core, API, HUD, provider prompts, voice, tools, or runtime adapters.

## Purpose

Resolver-to-executor integration defines how correction and delete commands without an explicit memory id may later become safe pending actions.

The integration must prevent guessing.

The integration must never execute memory changes directly from resolver output.

## Current boundary

MemoryCommandExecutor currently prepares correction and delete actions only when MemoryCommand.memory_id is present.

MemoryTargetResolver currently resolves explicit ids and bounded content-query matches into resolution outcomes.

These components are intentionally not connected yet.

## Future integration flow

Future resolver-to-executor flow should be:

- parse user text into MemoryCommand
- reject non-memory commands
- for correction or delete with explicit memory_id, prepare pending action directly
- for correction or delete without memory_id, collect a bounded candidate list
- call MemoryTargetResolver with the user query and bounded candidates
- handle no_match by returning a user-facing no-match result
- handle multiple_candidates by returning bounded candidates for user selection
- handle single_safe_match by creating a pending action with selected_memory_id
- require confirmation before any correction or delete operation executes

## Resolution outcome handling

no_match must not create a pending action.

multiple_candidates must not create an executable pending action.

unsafe_ambiguous_match must not create an executable pending action.

explicit_id_match may create a pending correction or delete action.

single_safe_match may create a pending correction or delete action.

## Candidate source rule

Candidate lists must be bounded before resolver execution.

Candidate lists may come from:

- MemoryService.list_memories with max_results bounds
- a previous bounded search result
- a user-selected result list

Candidate lists must not be generated by dumping all memory into provider prompts.

## User selection rule

When multiple candidates exist, the user must choose one candidate explicitly.

The choice must bind to exactly one memory id.

The selected memory id may then be used to create a pending correction or delete action.

Selection must not execute memory changes immediately.

## Confirmation rule

Correction and delete operations must still go through PendingMemoryActionService.

Confirmation must be bound to one concrete pending action.

Confirming a resolver-produced pending action must still call MemoryService with confirmed=True.

Cancel must clear the pending action without changing memory.

Stale pending actions must not execute.

## Safety boundaries

Resolver-to-executor integration must not:

- bypass MemoryService
- bypass MemoryPolicy
- bypass MemoryAuditSink
- inject memory into provider prompts
- call OS tools
- call external providers
- mutate MemoryItem objects
- store normal conversation automatically

## Non-goals

Milestone 112 must not add:

- resolver-to-executor implementation
- IntentRouter integration
- Core integration
- API endpoints
- HUD memory panels
- provider prompt injection
- automatic memory extraction
- semantic/vector search
- tag matching
- metadata matching
- OS/tool actions
- voice integration

## Next milestone direction

The next safe step is a small isolated resolver-to-executor plan/result model.

That model should describe whether resolver output should reject, ask user selection, or prepare a pending action.

Implementation should remain isolated from Core, API, HUD, providers, tools and runtime.
